### Task: Implement High-Throughput Click Tracking for URL Redirect Service

We are building a **URL shortener redirect service** using **Node.js, Fastify, Redis, MongoDB, and Socket.IO**.

The service must support **100,000 clicks per second without failures**.
Redirect latency target is **< 40ms**.

The redirect path must remain extremely lightweight.

---

### Current Stack

* Node.js (Fastify)
* Redis (cache + counters)
* MongoDB (persistent storage)
* Socket.IO (real-time dashboard)
* Mongoose
* TypeScript

---

### Redirect Flow Requirements

1. When a request comes for `/shortCode`, resolve the long URL using this order:

Memory Cache (LRU)
→ Redis cache (`url:{shortCode}`)
→ MongoDB fallback

2. MongoDB should **only be used when Redis cache misses**.

3. After resolving the URL, increment click count using Redis:

`INCR click:{shortCode}`

This must be **non-blocking and extremely fast**.

---

### Real-Time Dashboard

After incrementing the Redis counter:

Emit a Socket.IO event:

```ts
socket.emit("click_update", {
  shortCode,
  count
})
```

The dashboard listens to this event and updates the click count.

---

### MongoDB Persistence Strategy

MongoDB must NOT be updated per click.

Instead implement a **background worker** that runs every 60 seconds.

The worker must:

1. Scan Redis for keys matching `click:*`
2. Read the counters
3. Perform a MongoDB `bulkWrite` to increment `clickCount`
4. Delete or reset the Redis counters after successful persistence

Example MongoDB update:

```ts
{
  updateOne: {
    filter: { shortCode },
    update: { $inc: { clickCount: count } }
  }
}
```

---

### Performance Constraints

* No database writes inside redirect path
* Redis operations must be O(1)
* Mongo writes must use `bulkWrite`
* Worker must handle thousands of keys safely
* Avoid blocking the event loop

---

### Implementation Requirements

Implement:

1. `resolveUrl(shortCode: string)` service
2. Redis click counter increment
3. Socket.IO event emission
4. Background worker `flushClicksToMongo()`
5. Efficient Redis scanning (`SCAN`, not `KEYS`)

Use TypeScript and production-safe patterns.

The solution must prioritize **throughput, simplicity, and reliability**.
